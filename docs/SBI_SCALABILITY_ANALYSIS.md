# SBI 扩展性分析：从 200 到 10,000 包

## 当前架构

| 指标 | 值 |
|------|-----|
| 包数量 | 198 |
| 指纹 JSON 体积 | 4.37 MB（磁盘 13 MB，含换行） |
| 每包平均体积 | 22.6 KB |
| 搜索复杂度 | O(N × slots)，线性扫描 |
| 运行环境 | GitHub Pages，全量客户端下载 + 计算 |

### 指纹字段体积分布

| 字段 | 总体积 | 占比 | 权重 | 匹配使用 |
|------|--------|------|------|----------|
| splash_potion_variants | 1,824 KB | **41.4%** | — | **未接入**（生成但未引用） |
| apple_golden | 153 KB | 3.5% | **0.0** | 零贡献 |

| 包数 | JSON 体积（当前格式） | 比对量（相对 198 包） | 客户端下载时间* |

---

## 瓶颈分析

1. **下载瓶颈**：万包 = 220MB JSON，GitHub Pages 无服务端逻辑，用户必须全量下载
2. **计算瓶颈**：O(N) 遍历，每包多指标加权比对（dHash + moments + sig + hist + edge），中端手机不可行
3. **内存瓶颈**：220MB JSON 解析后 JS 对象更大，低端设备 OOM

---

## 精简方案（纯客户端，零成本）

### Phase 1：移除无效字段 → -54%

移除以下字段，零精度损失：

| 字段 | 原因 |
|------|------|
| `splash_potion_variants` | **41.4% 体积，sbi.js 未引用**，生成但从未接入匹配 |
| `apple_golden` | weight=0.0，零评分贡献 |
| `iron_sword` | 仅用于 UI 配色映射，不参与 slot 比对 |
| `xp_bar_bg` / `xp_bar_fill` | 不参与比对 |

结果：4.37 MB → 2.01 MB，万包 220 MB → 102 MB

### Phase 2：hist 量化 → -58%

当前 `hist` 为 72 个浮点数组（每项 6-8 字节），可量化为 Uint8（0-255），每项 1 字节。

结果：2.01 MB → 1.82 MB，万包 → 92 MB

> 精度风险：**极低**。hist 相似度用 `metricSimilarity`，量化误差 ±0.5/255 ≈ 0.2%，在 0.3 容差下无影响。

### Phase 3：分片加载 → 按需下载

将指纹拆为按类型分片：

| 分片 | 体积 | 典型搜索是否需要 |
|------|------|-----------------|
| diamond_sword.json | 146 KB | 是（最高权重） |
| ender_pearl.json | 150 KB | 是 |
| splash_potion.json | 153 KB | 是 |
| food.json (steak+gc) | 301 KB | 可选（权重 0.45） |
| widget.json | 56 KB | 是 |
| health.json | 399 KB | 是 |
| hunger.json | 428 KB | 是 |
| armor.json | 414 KB | 是 |

典型 PvP 截图（DS+EP+HL+HUD+Widget）需加载：~1.6 MB（Phase 1 格式），万包约 80 MB → 降至 ~41 MB

> 精度风险：**无**。分片不改变数据，只按需加载。food 分片跳过时 SK/GC 类 slot 不比对，但这些权重仅 0.45，对排名影响 < 2%。

### Phase 4：粗签名预过滤 → 跳过 70-90% 候选

用 DS/EP 的 `sig` 字段构建粗粒度桶：
- `darkFrac` (0-3) × `meanLum` (0-3) × `blueFrac` (0-3) × `G/B ratio` (0-3) = 256 桶
- 当前 198 包实测：DS 40 桶、EP 59 桶，平均每桶 3-5 包

搜索时：先对截图提取粗签名 → 定位桶 → 仅比对同桶 ± 1 的包。

> 精度风险：**低**。桶边界是连续特征的离散化，相邻桶包含几乎相同的签名。测试：需验证 9 张 test_img 仍全部 PASS。若某图 #1 落在桶边界，可扩展为 ±2 桶兜底。

### 综合效果

| 阶段 | 体积 | 万包体积 | 比对量 | 精度风险 |
|------|------|---------|--------|---------|
| 当前 | 4.37 MB | 220 MB | 100% | — |
| P1 移除无效 | 2.01 MB | 102 MB | 100% | 零 |
| P2 hist 量化 | 1.82 MB | 92 MB | 100% | 极低 |
| P3 分片加载 | ~1.6 MB* | ~41 MB* | 100% | 零 |
| P4 粗签名过滤 | ~1.6 MB | ~41 MB | **10-30%** | 低（需验证） |

\* 分片模式下只统计典型搜索实际下载量

**万包场景最终效果**：下载 ~41 MB，比对 ~1,000-3,000 包（而非 10,000），中端手机可接受。

## Cloudflare Workers 方案（低成本）

| 项目 | 说明 |
|------|------|
| 成本 | 免费额度 10 万次/天 / 付费 $5/月 50ms CPU |
| 架构 | 用户上传截图 → Worker 端比对 → 返回结果 |
| 数据存储 | Workers KV（免费 100KB/天写入，读取无限） |
| 万包可行性 | 需配合倒排索引；50ms CPU 对万包全量比对仍太紧 |
| 冷启动 | 10-50ms |
| 优势 | 零运维、全球 CDN |
| 劣势 | CPU 时间受限、KV 写入限制、冷启动 |

**结论**：配合 P4 倒排索引，付费 Workers 可支撑万包。但 VPS 性价比更优。

## VPS 方案（推荐，未来扩展）

| 项目 | 说明 |
|------|------|
| 推荐配置 | 1 vCPU + 1 GB RAM + 20 GB SSD |
| 成本 | Hetzner €3.5/月 / RackNerd $10/年促销 |
| 架构 | VPS 跑 SBI API（Node.js/Fastify），数据存内存/SQLite，材质包仍在 GitHub |
| 万包全量比对 | 单核 ~2-3s |
| 万包 + 倒排索引 | < 200ms |
| 客户端下载 | 0（只传结果列表，~2KB） |

### VPS 上的算法替代方案

| 方案 | 速度 | 精度 | 实现复杂度 | 说明 |
|------|------|------|-----------|------|
| 传统指纹 + 倒排 | O(log N), <50ms | 当前水平 | 低 | 在现有架构上加索引 |
| 特征嵌入 + FAISS | O(log N), <10ms | **更高** | 中 | 多维特征 → 128 维向量，ANN 检索 |
| 轻量 CLIP (ONNX) | O(N), ~500ms | 域外泛化差 | 高 | 已有 sbi-worker.js 基础，但 Minecraft 材质领域性太强 |
| 混合：嵌入初筛 + 传统精排 | O(log N) + O(簇内) | **最高** | 中高 | 先 ANN 取 top-50，再用传统指纹精排 |

**推荐**：VPS + FAISS + 传统特征嵌入化（不依赖 CLIP）。万包 < 50ms，成本 €3.5/月。

## 实施优先级

1. **P1 移除无效字段** — 立即可做，零风险，省 54%
2. **P3 分片加载** — 下一步，需改 sbi.js 的 fetch 逻辑
3. **P2 hist 量化** — 配合 P3 一起做
4. **P4 粗签名过滤** — 包数超 1000 时启用
5. **VPS + 嵌入** — 包数超 3000 时启用
